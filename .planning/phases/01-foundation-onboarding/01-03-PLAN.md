  <name>Drizzle schema with all 5 tables and inline RLS policies</name>
  <files>

  </files>
  <behavior>
    - households table: id (uuid PK defaultRandom), name (text notNull), created_at (timestamp defaultNow)
    - household_members table: id (uuid PK), household_id (FK→households onDelete:cascade), user_id (uuid notNull — references auth.users via sql, not drizzle FK), role (text notNull default 'member'), display_name (text), avatar_url (text nullable), joined_at (timestamptz defaultNow)
    - household_settings table: id (uuid PK), household_id (FK→households unique onDelete:cascade), household_type (text notNull — valid: couple|family_with_kids|flatmates|single), active_modules (text[] notNull default []), updated_at (timestamptz defaultNow)
    - household_invites table: id (uuid PK), household_id (FK→households onDelete:cascade), token (uuid unique defaultRandom), email (text nullable), invited_by (uuid notNull), expires_at (timestamptz notNull), claimed_at (timestamptz nullable), claimed_by (uuid nullable)
    - activity_feed table: id (uuid PK), household_id (FK→households onDelete:cascade), actor_id (uuid notNull), event_type (text notNull), entity_type (text notNull), entity_id (uuid nullable), metadata (jsonb), created_at (timestamptz defaultNow)
    - RLS SELECT policy on household_members: user can select rows where household_id IN (SELECT household_id FROM household_members WHERE user_id = auth.uid())
    - RLS INSERT policy on household_members: user can only insert rows where user_id = auth.uid()
    - RLS DELETE policy on household_members (admin-only): can delete where household_id IN (SELECT household_id FROM household_members WHERE user_id = auth.uid() AND role = 'admin')
    - RLS SELECT+INSERT policy on household_settings, household_invites, activity_feed: same household_id membership check pattern
    - RLS UPDATE policy on household_settings: any member can update (for module toggle)
    - households table: SELECT policy for members, INSERT for anyone authenticated (they'll immediately add themselves as admin)
  </behavior>
  <action>
    Create src/lib/db/schema.ts with all 5 tables. Key implementation details:

    1. For user_id referencing auth.users — Drizzle cannot cross-schema FK to auth.users directly. Use:
       user_id: uuid('user_id').notNull()
       and document with a comment: // References auth.users(id) — not a Drizzle FK (cross-schema)

    2. For household_type, use text() with Drizzle check constraint or simply text() — validation is at the application layer (Zod). Do NOT use pgEnum (adds migration complexity).

    3. For active_modules text[] in Drizzle use:
       activeModules: text('active_modules').array().notNull().default(sql`'{}'::text[]`)

    4. For metadata jsonb use:
       metadata: jsonb('metadata')

    5. Every table MUST have RLS policies defined via pgPolicy in the table's third argument array — following the exact pattern from ARCHITECTURE.md and 01-RESEARCH.md Pattern 2.

    6. The household_invites table needs NO update policy — token claim is done via service role (admin client) to prevent race conditions (KEY CONSTRAINT: atomic UPDATE WHERE claimed_at IS NULL).

    Create src/lib/db/index.ts:
    - Import postgres from 'postgres' and drizzle from 'drizzle-orm/postgres-js'
    - Use DATABASE_URL env var
    - Export db as the Drizzle instance
    - Add { prepare: false } for connection pooler compatibility

    Create drizzle.config.ts:
    - schema: "./src/lib/db/schema.ts"
    - out: "./src/lib/db/migrations"
    - dialect: "postgresql"
    - dbCredentials: { url: process.env.DATABASE_URL! }
  </action>
