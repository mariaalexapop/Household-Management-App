## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Compact element spacing within a row, between icon and label |
| md | 16px | Default element spacing — form field gaps, card internal padding |
| lg | 24px | Section padding — page content areas, card padding |
| xl | 32px | Layout gaps — between task list sections |
| 2xl | 48px | Major section breaks — between page header and list |
| 3xl | 64px | Page-level vertical spacing |

Exceptions: Touch targets (status toggle button, task checkbox) must be minimum 44px tall. Use `min-h-[44px]` with centered content to satisfy mobile tap target requirements per PLAT-03.

Source: standard 8-point scale; touch target exception drawn from PLAT-03 mobile requirement.

## Typography

| Role | Font | Size | Weight | Line Height | Usage |
|------|------|------|--------|-------------|-------|
| Body | Public Sans (`font-body`) | 16px | 400 (regular) | 1.5 | Task notes, descriptions, form helper text |
| Label | Public Sans (`font-body`) | 14px | 400 (regular) | 1.4 | Area badge text, filter labels, form field labels, metadata (due date, assignee) |

| Display | Plus Jakarta Sans (`font-display`) | 28px | 600 (semibold) | 1.2 | Chores dashboard card section header — not used in task list |

Rules (from DESIGN.md):
- Never use pure `#000000`; use `--color-kinship-on-surface` (`#2d3337`) for all body text.
- Pair a semibold heading with regular body to create the "Editorial Gap."
- Heading font is `font-display`; body/label font is `font-body`.

Source: layout.tsx (font definitions), DESIGN.md (typography rules), stitch/DESIGN.md.

## Color

| Role | Token | Value | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `--color-kinship-surface` | `#f7f9fb` | Page background (`bg-background`) |
| Secondary (30%) | `--color-kinship-surface-container-lowest` | `#ffffff` | Task row cards, form containers, notification dropdown |
| Secondary alt | `--color-kinship-surface-container` | `#eaeef1` | Page section backgrounds, filter bar background |
| Accent (10%) | `--color-kinship-primary` | `#0053dc` | Specific elements only — see below |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` | Delete task confirmation button only |
| Muted text | `--muted-foreground` | `oklch(0.556 0 0)` | Secondary metadata (area name when no area set, empty state body text) |

Accent (`#0053dc`) reserved for:
1. Primary CTA button — "Add Task" button fill
2. "Mark complete" checkbox/button — active state fill
3. Status badge `In Progress` — border and text color
4. Notification bell unread count badge — background
5. Active filter selection indicator
6. Chores module icon in header/breadcrumb

Do not use accent on: borders between list items, general hover states, informational labels.

"No-Line Rule" (from DESIGN.md): 1px solid borders are prohibited for sectioning. Use background color shifts instead:
- `surface` (`#f7f9fb`) as global page canvas
- `surface-container-lowest` (`#ffffff`) cards floating on top create natural boundary
- Task list rows: use `spacing-4` (16px) vertical whitespace to separate rows — no dividers

Source: CONTEXT.md (chores module colour = `#0053dc`), globals.css (all tokens), DESIGN.md (no-line rule, surface hierarchy, accent usage rules).

## Interaction Contract

### Task creation flow

1. User clicks "Add Task" — opens Dialog (base-ui/dialog, full form)
2. Form submits via Server Action (`createTask` in `src/app/actions/tasks.ts`)
3. On success: dialog closes, task row appears optimistically in list (sorted by starts_at)
4. On recurring: Inngest background function generates occurrences; parent task row not shown
5. On error: inline form error shown below the relevant field (not toast)

### Mark task complete

1. User clicks checkbox on task row
2. Optimistic UI: row status badge → "Done" immediately, row opacity 60%
3. Server Action `updateTaskStatus` fires
4. If "Hide done" is active: row fades out after 300ms transition
5. On error: revert optimistic state, show inline error in row

### Delete task

1. User opens row action menu ("..." / kebab)
2. "Delete task" option shown in red (`--destructive` color)
3. Confirmation: inline Dialog with copy — see Copywriting Contract
4. On confirm: Server Action `deleteTask` fires; row removed optimistically
5. On error: revert, show toast error

### Edit recurring task

1. User opens edit form for an occurrence
2. After form change, submit shows confirmation prompt: "Update all future occurrences?" — single option in Phase 2 ("this occurrence only" is Phase 6)
3. Server Action updates all future (not-done) occurrences

### Notification bell interaction

1. Bell shows unread count badge when `unreadCount > 0`
2. Click opens `NotificationDropdown` — popover/dropdown anchored to bell
3. Each notification row: message text + relative time ("2 minutes ago")
4. On open: Server Action marks all visible notifications as read (`readAt = now()`)
5. "No notifications" empty state if list is empty

### Filter interactions

All filters are stateful but URL-reflected (URL params) so the filtered view is shareable/bookmarkable. Filter changes trigger server-side re-fetch (no client-side JS filtering). No separate filter route.

## Accessibility Contract

- All interactive task row controls are keyboard-navigable (Tab order: checkbox → task name link → action menu)
- Status badges use `aria-label` in addition to visual color to convey status
- Notification bell: `aria-label="Notifications, {N} unread"` when unread count > 0; `aria-label="Notifications"` when zero
- Checkbox for mark-complete: `aria-label="Mark [Task name] complete"` / `aria-checked` state
- Delete confirmation Dialog: focus is trapped inside dialog; `Escape` closes dialog (cancels)
- Color is never the sole indicator: status badges always include text label alongside color
- Touch targets: minimum 44px height on all interactive row elements
