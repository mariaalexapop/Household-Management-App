- householdId: uuid FK -> households.id ON DELETE CASCADE
- make: text NOT NULL
- model: text NOT NULL

- plate: text NOT NULL
- colour: text (nullable)

**Table 2: `serviceRecords`**
- id: uuid PK defaultRandom
- householdId: uuid FK -> households.id ON DELETE CASCADE

- costCents: integer (nullable) -- COST-01: cost in cents

**Table 3: `insurancePolicies`**

2. After generation, manually append the following SQL to the generated migration file to create the Supabase Storage bucket and its RLS policy:

```sql
-- Create private storage bucket for household documents (PDFs)
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'documents',
  'documents',
  false,
  10485760,  -- 10MB
  ARRAY['application/pdf']
)
ON CONFLICT (id) DO NOTHING;

-- RLS policy: household members can manage their own household's files
CREATE POLICY "household_members_manage_docs" ON storage.objects
  FOR ALL TO authenticated
  USING (
    bucket_id = 'documents'
    AND (storage.foldername(name))[1] IN (
      SELECT household_id::text FROM household_members WHERE user_id = auth.uid()
    )
  )
  WITH CHECK (
    bucket_id = 'documents'
    AND (storage.foldername(name))[1] IN (
      SELECT household_id::text FROM household_members WHERE user_id = auth.uid()
    )
  );
```

3. Run `rtk npx drizzle-kit push` to apply the migration to the Supabase database.

4. Verify the tables were created by running a quick query (e.g., via the Supabase dashboard or `npx supabase db dump --schema public | grep -c 'CREATE TABLE'`).

IMPORTANT: If drizzle-kit push fails on the storage bucket SQL (since it's not a Drizzle-managed migration), run that SQL separately via the Supabase dashboard SQL editor or `npx supabase db execute`. The bucket creation SQL is infra setup, not schema migration.
